JV Partner Onboarding Flow
Click each step to expand details. This is what happens when a new restaurant partner connects their QuickBooks.

Admin Navigate to QuickBooks Settings click to expand

Admin logs into POS → Admin Panel → QuickBooks tab
URL: http://localhost:8090/admin/quickbooks
Auth: JWT Bearer token (admin role required)
What they see: 4 tabs — Connection, Account Setup, Mappings, Sync
Connection Tab shows: "Not Connected" status, "Connect to QuickBooks" button

System Generate OAuth Authorization URL

Frontend calls GET /integrations/quickbooks/connect
Backend does:
1. Generate CSRF state token (secrets.token_urlsafe(32))
2. Store state in memory with tenant_id + user_id (10-min TTL)
3. Build Intuit OAuth URL with: client_id, scope=com.intuit.quickbooks.accounting, redirect_uri, state
4. Return URL to frontend

Frontend does: Opens URL in popup window (600x700)

JV Partner Authorize on Intuit's OAuth Page

Partner logs into their Intuit account, selects their QB company, clicks "Allow"
This happens entirely on Intuit's servers. We never see their Intuit password.

What the partner sees:
1. Intuit login page (email + password)
2. Company selector (if they have multiple QB companies)
3. Permission consent screen: "POS System wants to access your QuickBooks"
4. Click "Connect" / "Allow"

Intuit then redirects to our callback URL with:
?code=AUTH_CODE&state=CSRF_TOKEN&realmId=QB_COMPANY_ID

System Exchange Code for Tokens + Save Connection

Backend callback validates state, exchanges auth code, encrypts tokens, stores in DB
Step by step:
1. Validate CSRF state (single-use, expires in 10 min)
2. POST to Intuit token endpoint with Basic auth (client_id:client_secret)
3. Receive: access_token (~60 min), refresh_token (~100 days rolling)
4. Encrypt both tokens with Fernet (AES-128, key derived from SECRET_KEY via SHA-256)
5. Fetch company info from QB API (company name, country, industry)
6. Upsert into qb_connections table with tenant_id + realm_id
7. Redirect popup to /admin/quickbooks?connected=1

DB record created:
QBConnection(tenant_id, realm_id, company_name, access_token_encrypted, refresh_token_encrypted, is_active=True)

Admin Run Account Matching

Account Setup tab → "Run Matching" → system fuzzy-matches 19 POS needs against partner's QB Chart of Accounts
What happens:
1. System fetches partner's full QB Chart of Accounts via API (typically 30-120 accounts)
2. For each of 19 POS accounting needs, runs 6-signal fuzzy matching:
  • Exact name match (1.0 weight if hit)
  • Anchor token match (0.40) — distinctive keywords after stripping accounting noise
  • Synonym overlap (0.25) — "gst" ↔ "tax", "bijli" ↔ "electricity"
  • QB AccountType compatibility (0.20)
  • Jaccard word overlap (0.10)
  • Substring containment (0.05)

3. Also tries each search hint (8-12 per need) for broader matching
4. Results: matched (≥0.80), candidates (0.55-0.79), or unmatched (<0.55)
5. Computes overall grade: A/B/C/F based on coverage percentage

Admin Review Matches & Make Decisions

For each POS need: accept auto-match, pick from candidates, create new QB account, or skip
Three decision types per need:

use_existing — Accept the auto-matched or manually-selected QB account
  Example: "Food Sales Income" → matched "Food Sales" at 1.000 → Accept

create_new — No good match exists, create a new QB account in partner's company
  Example: "Rounding Adjustment" not found → Create it in QB with correct AccountType

skip — This POS need doesn't apply to this partner
  Example: "Mobile Wallet" skipped if partner only accepts cash/card

All decisions must be made before applying. Required needs cannot be skipped.

System Apply Decisions → Create Mappings

Creates QB accounts (if needed) + writes qb_account_mappings rows in database
For each "use_existing" decision:
→ Insert QBAccountMapping(mapping_type, qb_account_id, qb_account_name, is_default=True)

For each "create_new" decision:
→ Call QB API: POST /v3/company/{realm_id}/account with correct AccountType + SubType
→ Get back new QB account ID
→ Insert mapping row with new account

For each "skip":
→ No mapping created, logged in result

Result: All POS accounting needs are now mapped to specific QB accounts. Sync is ready.

System Ongoing Sync: Orders → QB SalesReceipts

Every completed POS order becomes a SalesReceipt in partner's QB, using the mapped accounts
Trigger: Order status changes to "completed"
Pipeline:
1. on_order_completed() orchestrator fires
2. Pre-syncs QB Items (menu items) + Tax Codes if not yet synced
3. Builds SalesReceipt payload:
  • Line items with ItemRef (QB Item) + ItemAccountRef (mapped income account)
  • TxnTaxDetail with TaxCodeRef per line
  • DepositToAccountRef (mapped bank/cash account)
  • CustomerRef (if customer exists)
4. Enqueues as QBSyncJob(job_type="create_sales_receipt", idempotency_key=...)
5. Job processor POSTs to QB API
6. Success/failure logged to qb_sync_log

Retry: Failed jobs retry 3x with exponential backoff. Dead-letter queue for persistent failures.
Multi-JV-Partner Architecture
Each partner gets a completely isolated QuickBooks integration. One POS app registration, many partners.
Key Architecture: One Intuit App, Many Partners We register ONE Intuit Developer App (our POS). Each JV partner authorizes this app to access THEIR QB company via OAuth. Every database record is scoped by tenant_id — complete data isolation between partners.

POS System (Our App)

  • 1 Intuit Developer App registration
  • 1 QB_CLIENT_ID + QB_CLIENT_SECRET
  • 1 OAuth redirect_uri
  • Multi-tenant backend (FastAPI)
  • Shared fuzzy matching engine
  • Shared sync pipeline

Per Partner (Isolated)

  • Own QBConnection row (encrypted tokens)
  • Own account mappings (19 POS needs)
  • Own entity mappings (menu items → QB Items)
  • Own sync queue (jobs + retries)
  • Own audit trail (sync_log)
  • Own match results history

Intuit / QuickBooks

  • Each partner has own QB Online subscription
  • Each has own Chart of Accounts
  • Each has own realm_id (company ID)
  • OAuth tokens are per-company
  • API rate limits are per-company

Database Isolation Model
-- Every QB table is tenant-scoped. Example queries: -- Get Partner A's connection: SELECT * FROM qb_connections WHERE tenant_id = 'partner-a-uuid' AND is_active = true; -- Get Partner B's mappings (completely separate): SELECT * FROM qb_account_mappings WHERE tenant_id = 'partner-b-uuid'; -- Partner A can NEVER see Partner B's data: -- Auth middleware injects tenant_id from JWT automatically -- All queries filter by it -- Unique constraints prevent cross-contamination: UNIQUE(tenant_id, realm_id) -- 1 QB company per partner UNIQUE(connection_id, mapping_type, -- No duplicate mappings qb_account_name) UNIQUE(tenant_id, idempotency_key) -- No duplicate syncs
OAuth 2.0 Sequence
Detailed token lifecycle: authorization, encryption, refresh, revocation.

Token Security

# Token encryption at rest (oauth.py) SECRET_KEY → SHA-256 → 32 bytes → base64 → Fernet key Fernet.encrypt(access_token) → DB column (ciphertext) Fernet.decrypt(ciphertext) → plaintext token (in-memory only) # AES-128-CBC + HMAC-SHA256 (tamper-proof) # All workers with same SECRET_KEY can decrypt # DB dump without SECRET_KEY = useless encrypted blobs

Token Lifetimes

Access Token~60 minutes
Refresh Token~100 days (rolling)
CSRF State10 minutes
Auto-Refresh5 min before expiry
Refresh Flow (Automatic)
# Before EVERY QB API call: async def ensure_valid_token(connection, db): if connection.access_token_expires_at < now() + 5_minutes: connection = await refresh_access_token(connection, db) # POST to Intuit with refresh_token # Get new access_token + new refresh_token # Encrypt both, update DB # Rolling: refresh_token expiry extends 100 days from NOW return decrypt_token(connection.access_token_encrypted) # If refresh_token itself is expired (>100 days no activity): # → Set is_active = False # → Admin must re-authorize via OAuth popup
What Happens If Partner Doesn't Use POS for 100+ Days? The refresh token expires. Next time they try to sync, the system detects is_active = False and prompts them to re-connect via OAuth. All existing mappings are preserved — they just need to re-authorize. No data loss.
Fuzzy Matching Engine
How POS accounting needs are matched against a partner's QB Chart of Accounts.
Attempt 2 Innovation: No Templates Instead of picking from 40 pre-built templates (Attempt 1), the POS declares 19 fixed accounting needs and matches them directly against whatever accounts exist in the partner's QB. Works with any industry, any naming convention, any language mix (English + Urdu).

19 POS Accounting Needs

  • 7 Required: income, cogs, tax_payable, bank, discount, rounding, cash_over_short
  • 12 Optional: beverage_income, cash, mobile_wallet, tips, service_charge, delivery_fee, foodpanda_commission, gift_card_liability, rent, salary, utility, packaging

6 Matching Signals

  • Exact name match (1.00 — instant hit)
  • Anchor token match (0.40 — distinctive keywords)
  • Synonym overlap (0.25 — domain terms)
  • AccountType compat (0.20 — QB type check)
  • Jaccard word overlap (0.10 — raw overlap)
  • Substring containment (0.05 — partial)

Thresholds

  • ≥0.80 → HIGH confidence (auto-decide)
  • 0.55-0.79 → MEDIUM (candidate, review)
  • <0.55 → LOW (unmatched)

  • 35 synonym sets (incl. Urdu terms)
  • 12 accounting stop words stripped
  • Bigram matching for multi-word terms

Sync Pipeline Architecture
How POS orders flow into QuickBooks as SalesReceipts.

Order Completed in POS

Cashier marks order as "completed" (served + paid)
Trigger: PATCH /orders/{id} with status: "completed"
Order service fires on_order_completed(order_id) event.

Orchestrator Pre-Syncs Dependencies

Ensures QB Items (menu items) and Tax Codes exist before creating receipt
For each menu item in order:
• Check if QBEntityMapping(entity_type="menu_item") exists
• If not: create QB Item via API + store entity mapping

For each tax rate:
• Check if QBEntityMapping(entity_type="tax_code") exists
• If not: query/create QB TaxCode

Why pre-sync? QB SalesReceipt lines require ItemRef + TaxCodeRef. Without these, QB ignores ItemAccountRef and defaults income to "Services".

Build SalesReceipt Payload

Maps POS order data to QB SalesReceipt JSON using account mappings
Key mapping lookups:
• income mapping → ItemAccountRef (where income lands)
• bank or cash mapping → DepositToAccountRef (where money goes)
• discount mapping → Discount line item account
• tax_payable mapping → TxnTaxDetail
• US companies: TaxCodeRef: "TAX"/"NON", Intl: numeric TaxCode Id
• ShipAddr from company info (required for US AST tax calculation)

Enqueue Sync Job (Idempotent)

QBSyncJob created with idempotency_key to prevent duplicates
Idempotency: Key = create_sales_receipt:{order_id}
If job already exists for this order, skip (no duplicate QB receipts).

Priority: 5 (normal) for regular orders, 0 (critical) for retries
Max retries: 3 with exponential backoff
Dead letter: After 3 failures, moved to dead_letter status for manual review

QB API Call + Audit Log

POST /v3/company/{realm_id}/salesreceipt → success/failure logged
On success:
• Job status → completed
• QBSyncLog entry with full request/response, duration, QB doc number
• Connection's last_sync_at + last_sync_status updated

On failure:
• Job status → failed, retry_count++
• next_retry_at = now + (2^retry_count * 30 seconds)
• Error logged with QB error code + message
• After 3 failures: dead_letter status, admin notified
Production Readiness Gaps
Issues to address before going live with real JV partners. Demo-ready today, production needs these fixes.
!

In-Memory OAuth State Store

CSRF state tokens stored in Python dict. Breaks with multiple workers or server restarts. Fix: Move to Redis with 10-min TTL keys.

!

Token Refresh Race Condition

Two concurrent API calls can both try to refresh the same token simultaneously. Fix: Add SELECT ... FOR UPDATE lock on connection row during refresh.

Δ

No Intuit Webhook Handler

webhook_verifier_token column exists but no endpoint to receive token revocation events from Intuit. If partner changes QB password, we don't know until next API call fails.

Δ

OAuth Popup UX

After Intuit authorization, popup redirects but doesn't auto-close. User must manually close it. Fix: Add window.close() in callback redirect page or use postMessage.

Δ

No OAuth Event Audit Trail

Connect/disconnect/refresh events not logged to qb_sync_log. Hard to debug "when did they connect?" forensic questions.

i

Redirect URI Hardcoded for Dev

Currently http://localhost:8090/.... Production needs https:// + real domain. Already configurable via .env, just needs to be set correctly per environment.

i

Single QB Company Per Tenant

DB constraint UNIQUE(tenant_id, realm_id) allows multiple companies per tenant, but code's get_connection() returns first active one. Fine for current scope — needs connection selector UI if partners have multiple QB companies.

Bottom Line The integration is demo-ready and functionally complete. The gaps above are production-hardening items, not missing features. The core OAuth flow, fuzzy matching, account mapping, and sync pipeline all work end-to-end against real QB Sandbox data.
